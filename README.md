# Sahab System - Request & Approval Module

نظام "سحاب" هو وحدة مصغرة (Module) لنظام إدارة موارد مؤسسي (ERP)، تم تطويره باستخدام إطار عمل Laravel 12. يهدف النظام إلى إدارة تدفق الطلبات الداخلية (Workflow) وعمليات الاعتماد بين الموظفين والمدراء بشكل آمن ومنظم.

## التقنيات المستخدمة

- إطار العمل: Laravel 12 (MVC Architecture)
- قاعدة البيانات: MySQL
- المصادقة: Laravel Breeze (Session-based)
- الواجهات: Blade Templates + Tailwind CSS (تصميم Minimalist أبيض وأسود)
- المتطلبات الإضافية: PHP 8.2+, Composer, Node.js

## طريقة الإعداد والتشغيل (Setup)

لتشغيل المشروع على بيئتك المحلية، يرجى اتباع الخطوات التالية:

1. نسخ ملف إعدادات البيئة:

```bash
cp .env.example .env
```

2. تثبيت الاعتمادات (Dependencies):

```bash
composer install
npm install
npm run build
```

3. توليد مفتاح التطبيق:

```bash
php artisan key:generate
```

4. إعداد قاعدة البيانات:
   قم بإنشاء قاعدة بيانات MySQL فارغة، ثم قم بتحديث بيانات الاتصال في ملف `.env`:

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=sahab_exam
DB_USERNAME=root
DB_PASSWORD=
```

5. تشغيل التهيئة والبيانات التجريبية (Migrations & Seeders):
   (هذا الأمر سيقوم ببناء الجداول وإنشاء حسابات الاختبار تلقائياً)

```bash
php artisan migrate --seed
```

6. تشغيل الخادم المحلي:

```bash
php artisan serve
npm run dev
```

النظام متاح الآن على الرابط: http://localhost:8000

## سير العمل (Workflow)

- الموظف العادي: يقوم بتسجيل الدخول، ويستطيع إنشاء طلب جديد (العنوان والوصف). تظهر حالة الطلب كـ "Pending" ولا يرى سوى الطلبات التي قام بإنشائها بنفسه.
- المدير: يقوم بتسجيل الدخول، ويستطيع رؤية جميع الطلبات في النظام.
- الاعتماد والرفض: تظهر أزرار (موافقة / رفض) للمدير فقط بجانب الطلبات المعلقة.
- الإشعارات: عند قيام المدير بتحديث حالة الطلب، يتم تسجيل إشعار (Log Notification) تلقائياً في قاعدة البيانات يوضح الإجراء الذي تم، ويرتبط بالمستخدم صاحب الطلب.

## الصلاحيات والأمان (Authorization & Security)

تم تطبيق أفضل ممارسات الحماية في Laravel لضمان أمان النظام:

- استخدام Policies: تم بناء SystemRequestPolicy للتحكم في من يحق له رؤية الطلبات (view) ومن يحق له اعتمادها (manageStatus). تم تجنب الشروط المباشرة (Hardcoded logic) تماماً.
- استخدام Enums: تم استخدام PHP Enums (UserRole و RequestStatus) لتوحيد قيم الصلاحيات وحالات الطلب وضمان عدم إدخال قيم خاطئة.
- التحقق من البيانات (Form Requests): تم استخدام كلاس StoreSystemRequest لتنظيف والتحقق من صحة مدخلات المستخدم قبل إرسالها لقاعدة البيانات.

## واجهة برمجة التطبيقات (API Endpoint)

يوفر النظام نقطة وصول API محمية لجلب الطلبات:

- المسار: GET `/api/requests`
- الحماية: تتطلب تسجيل دخول صالح.
- الاستجابة: اذا كان المستخدم موظفاً، ترجع بيانات طلباته فقط. إذا كان المستخدم مديراً، ترجع جميع الطلبات المتاحة في النظام. يتم تنسيق الاستجابة باستخدام API Resource لضمان إرجاع JSON نظيف ومنظم.

## التحسينات المستقبلية (Future Improvements)

لو كان هذا النظام سيُطرح للإنتاج الفعلي (Production)، يمكن إضافة التحسينات التالية:

- نظام إشعارات حقيقي: تحويل تسجيل الإشعارات من مجرد سجل في قاعدة البيانات إلى إشعارات لحظية باستخدام (WebSockets/Pusher) أو إشعارات بريد إلكتروني.
- سجل التدقيق (Audit Trails): تتبع جميع التغييرات التي تحدث على الطلب (من قام بالتعديل ومتى) للحفاظ على شفافية عمليات الـ ERP.
- تصفية وتقسيم الصفحات (Pagination & Filtering): إضافة ميزة الترقيم للطلبات وخاصية البحث والفلترة حسب الحالة لتسهيل عرض البيانات عند زيادتها.
- تدفق موافقات متعدد المستويات (Multi-level Approvals): السماح بتمرير الطلب لعدة مدرّاء (مثلاً: مدير مباشر ثم مدير مالي) قبل الاعتماد النهائي.
